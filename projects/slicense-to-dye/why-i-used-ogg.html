Why I Used .ogg

The problem

As soon as I jumped into this project, I noticed that there was a problem with looping sounds. Even if I edited the sound file so the cutoff is at a perfect timing on a computer, there was a noticeable delay when we built the project and ran it on a mobile device. I did a little research. Turns out that phones have to load up the sound file every time before playing, even if that sound was just played. So I started thinking about the basic ways to solve this issue:

  • Always have the next song preloaded so that it's available right when the current song ends.
  • Break our song into separate parts so the load time for each one is less noticeable.

Preloading the next song

For most phones, this was a pretty good solution. However, for my low-end benchmark phone, weird things started happening. I supposed it was a problem of not having enough access memory to work with. The end result was a timing worse than what we originally had.


Breaking song into separate parts

This was highly impractical because we would have so many sound files. Like upwards to a hundred or so. But I tried it anyway, and the result was still –– unsurprisingly –– pretty bad. No matter what I did, there would always be a lingering delay in which the phone has to load the next song.


Then I stumbled upon .ogg

It turns out that .ogg loads a lot faster than the file types we were originally using (.wav and .mp3). There was still maybe a 0.1s delay, but it was a lot less noticeable than the delay we had before, which was approximately 0.5s. It seemed to solve the problem for our case, but I'm not too sure what would happen with much different file sizes.
